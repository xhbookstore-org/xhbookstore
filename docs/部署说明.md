# 自动构建与部署

## 发布流程

- 推送到 `main`：GitHub Actions 自动测试、构建并部署测试环境。
- 生产发布：在 GitHub Actions 的 `Build and Deploy` 工作流中选择 `Run workflow`，环境选择 `prod`。
- 后端由 Maven 完整执行测试和打包；前端按目标环境分别执行 `build:stage` 或 `build:prod`。
- GitHub Actions 将 JAR、UI ZIP、部署脚本和校验文件组成单一发布包，再通过 SSH 交给服务器安装。
- 数据库迁移不随代码自动执行，必须先备份并独立审批、执行和验证。

## 服务器发布机制

服务器脚本安装在 `/www/server/java/projects/deploy.sh`，发布版本保存在
`/www/server/java/projects/versions/<version>`。

每次发布会依次执行：

1. 校验 JAR、UI ZIP 和版本名。
2. 创建不可复用的版本目录。
3. 保存上一个 JAR 软链接和 UI 目录作为回滚点。
4. 原子切换 Admin/API JAR 软链接并替换 UI。
5. 重启 `spring_xhbookstore-admin` 和 `spring_xhbookstore-api`。
6. 检查服务状态以及 Admin/API 本机健康接口。
7. 任一步骤失败时恢复旧软链接和 UI，并重新启动旧版本。
8. 发布成功后默认保留最近 8 个版本。

## GitHub 配置

仓库 Actions Secrets：

- `SSH_PRIVATE_KEY`：同时具有测试、生产服务器最小部署权限的 SSH 私钥。
- `PROD_HOST`：生产服务器地址。

生产发布使用 GitHub `prod` Environment；建议在仓库设置中为该 Environment 配置审批人和仅允许 `main` 分支部署。

## 手工回滚

自动健康检查失败会自动回滚。若上线后需要人工回滚，应重新发布已验证的提交或发布包，不要直接覆盖当前 JAR。回滚后必须再次检查：

- JAR 软链接目标。
- Admin/API systemd 状态与 8090、8091 监听端口。
- Nginx 首页、`/prod-api/` 和 `/api/` 代理。
- 应用近期错误日志。
