# P02 — 积分发放与消耗需求

> 版本：V2.2
>
> 状态：新需求设计稿，待业务确认、待开发
>
> 关联文档：[P01-积分体系](P01-积分体系.md)、[P03-积分发放与消耗管理表设计](P03-积分发放与消耗管理表设计.md)
>
> 适用范围：积分规则、自动发放、人工录入、积分消耗、预算控制、会员协议

## 1. 设计结论

第一期不建设活动表、任务表和技术事件表，只新增一张 `xhbs_points_rule`，统一配置积分获取和消耗；实际操作继续写入并扩展现有 `xhbs_points_order`。

线下活动核销和付费活动抵扣直接创建自定义积分规则，填写规则名称、积分方向、额度、有效期和适用门店。该方案只管理积分，不管理活动报名、场次、宣传和参与名单。

推荐处理链路：

```text
注册/购卡/借阅/线下核销/人工消费单
    → 生成唯一业务键
    → 匹配生效中的积分规则
    → 校验规则状态、预算、权限、重复发放和批量采购标记
    → 创建积分订单及入账/出账批次
    → 更新会员积分余额
    → 记录处理结果；失败时重试或进入异常列表
```

## 2. 业务原则

### 2.1 经营目标

积分用于提高用户留存和会员粘性。会员通过注册、购卡、借阅、消费和活动获得积分，并用于活动权益或礼品兑换。

### 2.2 预算先行

积分发放规则和兑换比例发布前必须完成费用预算分析。

系统必须防止：

- 同一业务重复发放。
- 超过活动或规则预算继续发放。
- 人工录入超过岗位权限。
- 退款后奖励积分没有追回。
- 会员日倍增被重复计算。
- 通过拆单规避单笔或每日限额。

### 2.3 已发积分保护

积分一旦成功入账，不能直接编辑、删除或无理由收回。

允许减少积分的情形仅包括：

1. 用户主动兑换活动权益或礼品。
2. 用户使用积分抵扣付费活动。
3. 会员卡退订，追回该会员卡对应的奖励积分。
4. 商品或图书退货，追回原消费产生的积分。
5. 积分达到会员协议约定的一年有效期。
6. 明确的原业务取消或错误发放冲正，且有原业务单号和审批记录。

不能通过后台直接修改会员余额来降低已发行积分。经营成本需要调整时，应通过版本化的兑换规则提高未来兑换所需积分，但不能修改历史余额和已经完成的兑换订单。

### 2.4 有效期

每一笔入账积分有效期为一个自然年：

```text
expired_at = credited_at + 1 calendar year
```

系统按积分入账批次管理有效期，消耗时优先使用最早过期的积分。

## 4. 积分获取规则

### 4.1 初始规则

| 获取类型 | 触发方式 | 计算规则 | 当前状态 | 业务说明 |
|---|---|---:|---|---|
| 注册普通会员 | 自动 | 固定 100 | 基础业务已有，积分待接入 | 每个会员只发放一次 |
| 购买尊享会员 | 自动 | 固定 365 | 基础业务已有，积分待接入 | 销售单支付成功即触发 |
| 购买畅享会员 | 自动 | 固定 188 | 基础业务已有，积分待接入 | 销售单支付成功即触发 |
| 续费尊享会员 | 自动 | 固定 365 | 基础业务已有，积分待接入 | 续费销售单支付成功即触发 |
| 续费畅享会员 | 自动 | 固定 188 | 基础业务已有，积分待接入 | 续费销售单支付成功即触发 |
| 线下参与活动核销 | 自动 | 活动自定义 | 暂无 | 员工扫描会员动态码完成核销后触发 |
| 借阅图书 | 自动 | 每册 10 | 开发中 | 一次借 N 册获得 `N × 10` 积分 |
| 购买书籍 | 人工消费录入 | 实付 1 元=1积分 | 开发中 | 员工扫码会员码后录入消费 |
| 购买店内自营商品 | 人工消费录入 | 实付 1 元=1积分 | 开发中 | 员工扫码会员码后录入消费 |
| 购买储值卡 | 人工消费录入 | 实付 1 元=1积分 | 开发中 | 员工扫码会员码后录入消费 |

### 4.2 注册积分

触发事件：`MEMBER_REGISTERED`。

规则：

1. 微信自动注册和 Admin 新增普通会员均可触发。
2. 每个会员最多成功发放一次。
3. ERP 历史会员导入不补发注册积分，除非业务单独批准补发活动。
4. 政府或单位批量开通会员时不发注册积分。
5. 注册事务成功后写入积分事件；不能在登录 Controller 中直接修改积分。

### 4.3 购买和续费会员卡积分

触发事件：`MEMBER_CARD_ORDER_PAID`。

积分奖励绑定 `member_card_order.order_no`，不绑定会员卡实际激活时间。即使会员已有生效卡、新卡处于待生效队列，积分仍按销售单支付成功时间触发。

系统需要给会员卡类型增加稳定业务编码，不能根据中文名称或有效天数判断：

| 业务编码 | 会员类型 | 购买积分 | 续费积分 |
|---|---|---:|---:|
| `PREMIUM` | 尊享会员 | 365 | 365 |
| `ENJOY` | 畅享会员 | 188 | 188 |
| `NORMAL` | 普通会员 | 0 | 0 |

销售单增加 `order_type`：

- `OPEN`：购买/开通。
- `RENEW`：续费。

积分规则通过“会员类型业务编码 + 销售单类型”匹配，不依赖当前 `is_renewal` 中文卡类型拆分。

### 4.4 借阅积分

触发事件：`BORROW_ORDER_CREATED`。

规则：

1. 借阅订单和全部逐册明细成功提交后触发；这是业务服务调用，不需要新建活动表、任务表或技术事件表。
2. 借阅明细一条只代表一册书，图书编号和书名由员工手工填写；积分计算不读取 `book_info`、book history 或库存。
3. 借阅服务只传入 `sceneCode=BORROW_BOOK`、借阅单号和成功明细数 `baseQuantity`，不传入最终积分。
4. 积分服务查询当时生效的 `xhbs_points_rule`，初始配置为 `PER_ITEM` 每册 10 分；奖励积分=`baseQuantity × 规则每册积分`。
5. 同一借阅单号只能奖励一次，全局业务键为 `BORROW_BOOK:{borrowOrderNo}`。
6. 积分订单保存明细数、每册积分、规则和计算快照，后续不因规则修改而改写历史奖励。
7. 规则停用、过期或未匹配时，借阅可正常成功，但必须保存未发放原因以便排查或安全重试。
8. 借阅单在发放前被取消时不发积分；已发放后只有明确误录撤销才关联原积分订单冲正。
9. 正常归还、损坏归还、遗失或借转购不撤销借阅奖励。

### 4.5 消费录入积分

员工扫描会员动态码后创建“消费积分录入单”，员工录入的是消费信息，不是最终积分。

必填字段：

| 字段 | 说明 |
|---|---|
| 会员 ID | 扫描动态会员码获得 |
| 消费类型 | 图书、自营商品、储值卡 |
| 实付金额 | 积分计算依据 |
| 消费时间 | 判断会员日的依据 |
| 小票号/业务单号 | 防止重复录入 |
| 消费门店 | 当前员工门店 |
| 操作员工 | 当前登录员工 |
| 凭证图片 | 建议必传，至少高额录入必传 |
| 备注 | 异常或补录时必填 |

积分计算：

```text
基础积分 = floor(实付金额，单位：元)
最终积分 = 基础积分 × 活动倍数
```

例如实付 99.80 元，普通日获得 99 积分，双倍会员日获得 198 积分。

不允许员工手工修改系统计算出的最终积分；确需调整时使用独立人工调整流程并记录审批。

小票号在同一门店和消费类型下必须唯一，防止重复录入。没有小票号的业务必须提供系统业务单号或进入店长审批。

## 5. 会员日规则

初始会员日为每月 6 日、16 日、26 日。

| 规则 | 初始值 |
|---|---|
| 适用类型 | 购买图书、购买店内自营商品 |
| 不适用类型 | 购买储值卡、注册、购卡、借阅、活动积分 |
| 倍数 | 2 倍 |
| 判断时间 | 实际消费时间 |
| 时区 | `Asia/Shanghai` |

会员日必须做成可配置的促销活动，不要在代码中写死日期。配置支持：

- 每月指定日期。
- 固定日期范围。
- 指定门店。
- 指定消费类型。
- 积分倍数。
- 单会员上限、活动总预算。
- 启用、停用和版本生效时间。

补录消费时原则上只允许录入当天消费。店长可以在配置允许的补录期限内补录，系统仍按实际消费时间判断会员日，并保存补录原因。

最终积分超过操作员权限上限时，不能拆单规避，应提交店长处理或审批。

## 6. 积分消耗规则

### 6.1 初始规则

| 消耗类型 | 触发方式 | 消耗规则 | 当前状态 | 业务说明 |
|---|---|---:|---|---|
| 付费活动抵扣 | 自动 | 活动自定义 | 暂无 | 报名或确认参加时扣减 |
| 兑换礼品 | 人工兑换单 | 礼品自定义 | 暂无 | 员工扫码会员码，选择礼品后扣减 |
| 尊享会员卡退订 | 自动 | 365 | 基础业务已有，积分待接入 | 关联原销售单和退款单 |
| 畅享会员卡退订 | 自动 | 188 | 基础业务已有，积分待接入 | 关联原销售单和退款单 |
| 退书或商品 | 人工退货单 | 按原订单实发积分 | 开发中 | 必须关联原消费积分录入单 |
| 积分过期 | 自动任务 | 剩余未使用积分 | 待开发 | 入账满一年后自动处理 |

### 6.2 付费活动抵扣

活动创建时配置：

- 是否允许积分抵扣。
- 固定扣减积分或阶梯方案。
- 每人参与次数。
- 活动总名额和积分预算。
- 扣减时点：报名成功、支付确认或线下核销。
- 取消报名后的积分退回规则。

扣分前必须锁定会员积分账户并完成入账批次核销。活动取消或符合退款条件时，通过关联原积分订单的 `BK` 退回积分。

### 6.3 礼品兑换

不建议员工直接输入任意扣减积分。应创建礼品或权益兑换项：

| 字段 | 说明 |
|---|---|
| 礼品/权益名称 | 展示名称 |
| 所需积分 | 后台配置 |
| 可兑换库存 | 防止超兑 |
| 每人限兑 | 次数限制 |
| 开始、结束时间 | 生效范围 |
| 适用门店 | 可选 |
| 状态 | 草稿、已发布、已结束 |

员工扫码会员码后选择兑换项，系统创建兑换订单并自动扣分。若本期只做人工扣减，至少必须选择“礼品兑换”事项、填写礼品名称和凭证，不允许只填一个积分数字。

### 6.4 会员卡退订

退订时根据原销售单查找购卡奖励积分订单，并扣回对应积分：

- 尊享会员：365。
- 畅享会员：188。
- 政府批量采购未发过积分，无需扣回。
- 同一退款单只能处理一次。

退订积分必须和会员卡退款事务闭环：退款成功但积分处理失败时需要进入补偿任务，不能静默遗漏。

#### 已确认的冻结方案

- 尊享、畅享购卡和续费积分支付成功后立即入账并展示，但冻结 7 天，不进入可用余额。
- 冻结积分不能用于礼品兑换、活动抵扣、退货扣分或任何人工扣分。
- 7 天内退卡时创建 `BK` 反向订单并冲正原冻结批次，不扣会员其他可用积分。
- 退款窗口结束后，由定时任务将冻结批次转为可用，并把积分从冻结余额转入可用余额。
- 退款与解冻锁定同一销售订单和积分批次；退款允许条件为 `当前时间 < available_at`，解冻条件为 `当前时间 >= available_at`。
- 解冻任务必须幂等，重复执行不能重复增加可用积分。

### 6.5 退书或商品

退货必须关联原消费积分录入单或销售单，扣回原订单实际发放的积分，而不是按退货当天规则重新计算。

例如会员日在原订单获得双倍积分，退货也应按原订单的双倍积分比例扣回。

部分退货时按原规则快照计算，并且累计扣回积分不得超过原订单已发积分。无法找到原订单时进入店长审批，不能由店员任意输入扣分。

## 7. 积分有效期和会员协议

### 7.1 有效期计算

1. 每一笔入账形成独立积分批次。
2. 有效期为入账时间起一个自然年。
3. 消费积分采用“先过期先使用、同到期时间先进先出”。
4. 每日定时任务处理已经到期且仍有余额的批次。
5. 过期产生 `EX` 积分订单和核销明细，不能直接修改余额。
6. 过期任务必须幂等，重复执行不能重复扣减。

### 7.2 提醒

建议在到期前 30 天、7 天提醒会员，展示即将过期积分和到期日。提醒失败不能影响积分账务。

### 7.3 最新会员协议

会员协议必须更新并形成新版本，至少明确：

- 积分获取途径。
- 积分有效期为一年及具体计算方式。
- 积分兑换规则可能按新活动调整。
- 会员卡退订、商品退货时对应奖励积分的处理方式。
- 积分不可提现、不可转让等业务限制。
- 积分到期提醒方式及会员自行关注义务。

建议增加协议版本和同意记录：

```text
member_agreement
member_agreement_acceptance
```

新注册会员必须同意最新协议。老会员应展示协议更新通知并记录确认时间。

历史积分没有可靠入账日期时，不能直接按历史创建时间过期。建议把上线迁移日作为历史期初积分批次起点，在完成协议通知后一年到期；具体过渡规则需业务和法务确认。

## 8. 积分获取/消耗类型配置

新增 `xhbs_points_rule`，统一配置所有积分获取和消耗类型。详细字段以 P03 为准。

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `rule_code` | 稳定唯一编码 |
| `rule_name` | 获取或消耗类型名称 |
| `scene_code` | 注册、购卡、借阅、消费、核销、兑换或退货场景 |
| `direction` | `ADD`、`DEDUCT` |
| `trigger_mode` | `AUTO`、`MANUAL` |
| `trigger_event` | 自动规则对应的业务事件 |
| `calculation_mode` | `FIXED`、`PER_ITEM`、`PER_YUAN`、`MANUAL`、`ORIGINAL_ORDER` |
| `fixed_points` | 固定积分 |
| `points_per_unit` | 每册或每元积分 |
| `member_day_enabled/days/multiplier` | 会员日日期和倍数 |
| `member_limit` | 单会员周期上限 |
| `budget_points` | 规则总积分预算 |
| `require_biz_order` | 是否必须关联业务单 |
| `require_evidence` | 是否要求凭证 |
| `manual_points_editable` | 是否允许员工输入积分；金额类规则应为否 |
| `freeze_days` | 入账冻结天数；购卡和续费规则为 7 |
| `card_type_ids` | 适用会员卡类型 |
| `dept_ids` | 适用门店范围 |
| `effective_from/to` | 生效时间 |
| `status` | 草稿、启用、停用、结束 |
| `operator_user_id/name/time` | 最近操作人和操作时间 |

后台可勾选自动或手动，但应遵守：

- 注册、购卡、续费、退卡、借阅和过期只能配置为自动。
- 消费录入、礼品兑换和退货可配置为手动业务单。
- 自动规则必须配置唯一业务事件。
- 已启用规则不能覆盖核心计算字段；修改时停用旧规则并复制新规则。
- 历史积分订单保存规则编码、名称和计算参数快照。

## 9. 线下核销和付费活动的简化处理

第一期不建立活动领域模型。每一个需要独立名称、积分、有效期或门店范围的线下活动，直接在 `xhbs_points_rule` 中创建一条自定义规则。

线下核销流程：

1. 员工选择当前门店生效中的自定义积分规则。
2. 扫描会员动态码并校验有效期。
3. 校验规则有效期、会员资格、门店、次数和预算。
4. 在 `xhbs_points_order` 创建带唯一业务键的积分日志。
5. 使用日志状态记录待处理、成功或失败。
6. 自动增加或扣减积分。
7. 日志保存规则、计算参数、操作人和操作时间快照。

同一规则、会员和核销次数必须具备唯一业务键，防止重复扫码。

该方案不提供报名、场次、宣传页面和参与名单。如果后续确认需要，再独立设计活动业务。

## 10. 处理状态和失败重试

第一期不新增 `points_event` 或任务处理表。`xhbs_points_order` 增加业务键、幂等键和处理状态，直接承担积分操作日志和失败重试入口。

状态至少包括：待处理、处理中、成功、失败和已冲正。失败记录保存错误码和错误信息，后台只能安全重试或按原单冲正，不能直接编辑积分结果。

Quartz 只负责扫描超时待处理记录、失败重试、积分过期和对账，不为每种积分规则创建单独任务。

当前不建设每日签到、邀请、分享等通用会员任务中心。

## 11. 人工积分业务单

人工场景不能直接调用通用加减余额接口。建议新增 `points_manual_order`：

| 字段 | 说明 |
|---|---|
| 人工单号 | 唯一编号 |
| 会员 ID/编号/姓名 | 会员及快照 |
| 规则 ID、编码、版本 | 消费、礼品、退货等类型 |
| 方向 | 增加或扣减 |
| 消费金额 | 金额型规则使用 |
| 基础积分 | 计算前积分 |
| 活动 ID、倍数 | 会员日等活动快照 |
| 最终积分 | 实际变动值 |
| 原业务单号 | 退货时关联原单 |
| 小票号 | 防重复 |
| 消费/退货时间 | 业务时间 |
| 门店、员工 | 操作范围和审计 |
| 凭证图片 | 小票或礼品凭证 |
| 状态 | 待确认、成功、已冲正、已拒绝 |
| 审批信息 | 超限时使用 |
| 积分订单号 | 成功后的账务订单 |

人工单成功后再创建积分订单。人工单、积分订单和会员余额必须可双向追溯。

## 12. 批量采购不发积分

不能继续使用“会员备注中有单位名称”作为不给积分的判断依据。备注是展示信息，不是稳定业务字段。

建议增加：

| 表 | 字段 | 说明 |
|---|---|---|
| `member_card_order` | `is_bulk_purchase` | 本次会员卡是否政府/单位批量采购 |
| `member_card_order` | `bulk_purchase_batch_no` | 批量采购批次号 |
| `member_card` | `is_bulk_purchase` | 销售单标记快照 |
| `member` | `is_bulk_registration` | 首次注册是否来自批量开通 |
| `member` | `bulk_purchase_batch_no` | 首次批量开通来源批次 |

规则：

1. 批量开通时不发注册 100 积分。
2. 批量购买尊享/畅享会员卡时不发 365/188 积分。
3. 对应批量卡退订时也不扣回未发放积分。
4. 会员以后个人购买或续费会员卡时，按个人订单正常获得积分。
5. 是否发积分根据本次业务单标记判断，不根据会员永久身份或备注判断。

批量采购字段必须由批量开卡流程或有权限的后台员工明确设置，并记录操作日志。

## 13. 角色权限和额度

积分录入额度必须后台可配置，不在代码中写死角色名称。

建议新增 `points_operator_policy`：

| 字段 | 说明 |
|---|---|
| 角色 ID | 关联系统角色 |
| 是否允许人工增加 | 权限开关 |
| 是否允许人工扣减 | 权限开关 |
| 允许规则范围 | 消费、礼品、退货等 |
| 单笔最大积分 | 空表示不设业务上限 |
| 单日累计上限 | 建议配置，防止拆单 |
| 是否需要审批 | 超额或高风险场景 |
| 可操作门店范围 | 配合数据权限 |
| 生效时间、状态 | 配置管理 |

初始配置：

| 角色 | 人工积分权限 | 单笔最终积分上限 |
|---|---|---:|
| 店员 | 允许 | 3000 |
| 店长 | 允许 | 不设业务上限，但仍受规则和预算限制 |
| 经理 | 不允许 | — |

店员 3000 确认为单笔上限，不设置每日累计上限。按会员日倍增后的最终积分绝对值判断，增加和扣减都适用；超过 3000 时整单拒绝并由店长操作，不能拆分同一小票或业务单规避。

一个用户拥有多个角色时，采用其已启用策略中的最高可用额度；明确禁用的角色本身不授予权限。个人例外授权需要单独记录，不能修改角色名称来绕过策略。

自动积分不受员工单笔录入额度限制，但必须受积分规则预算限制。

## 14. 预算和防超发

第一期只建立规则预算，预算字段直接保存在 `xhbs_points_rule`。

### 14.1 预算配置

| 配置 | 说明 |
|---|---|
| 预算周期 | 年、季度、月或活动周期 |
| 预算积分 | 最大允许发放积分 |
| 单积分预计成本 | 用于估算财务负债 |
| 预算金额 | 最大预计成本 |
| 单会员上限 | 防止少数会员过度获取 |
| 预警比例 | 建议 80%、90%、100% |
| 超限策略 | 禁止新参与、停止规则或进入审批 |

### 14.2 原子预算占用

发积分前必须在数据库事务内原子占用预算：

```text
issued_points + 本次积分 <= budget_points
```

预算占用成功后才能创建积分订单。业务失败或积分订单回滚时同步释放预算。

### 14.3 已承诺权益保护

对已经发布并向会员承诺的固定权益，不能因为后台预算耗尽而静默拒绝会员已经完成条件后的积分。

系统应在预算不足前停用对应自定义规则、暂停相关产品或触发紧急预警；已经完成注册、购卡、借阅或核销条件的会员仍应按已发布规则入账。

### 14.4 预算看板

至少展示：

- 本周期已发积分。
- 已消费积分。
- 已过期积分。
- 当前未消费积分负债。
- 预计兑换成本。
- 各积分规则预算使用率。
- 即将超预算规则。
- 重复事件、失败事件和人工高额录入。

## 15. 后台菜单和页面

建议积分管理菜单：

```text
积分管理
├── 积分流水
├── 积分发放管理
├── 人工积分操作
├── 积分预算
└── 异常处理
```

会员协议可放在“协议管理”或系统内容管理中，不建议混入积分菜单。

### 15.1 积分发放管理

支持：

- 查询获取/消耗类型。
- 配置自动/手动。
- 配置固定、按册、按元、自定义计算。
- 配置有效期、预算、上下限和适用范围。
- 创建线下核销或付费活动自定义规则。
- 启用、停用、结束和复制规则。
- 查看最近操作人、操作时间和关联积分日志。

已启用规则不能直接覆盖核心计算字段，必须停用旧规则并复制新规则。

### 15.2 人工积分操作

支持按会员、手机号、小票号、类型、门店、员工、时间、状态查询。提供扫码录入、详情、凭证预览、审批和关联积分流水。

### 15.3 异常处理

展示处理失败、预算不足、重复业务事件、积分余额不一致和退款追回失败记录。只能补处理或生成冲正，不能直接编辑余额。

## 17. 幂等、事务和一致性

1. 所有自动事件以业务单号建立唯一约束。
2. 注册积分以会员 ID 和规则编码唯一。
3. 购卡积分以销售单号和规则编码唯一。
4. 退卡扣分以退款单号唯一，并关联原销售积分订单。
5. 借阅积分以借阅单号唯一。
6. 人工消费以门店、消费类型和小票号排重。
7. 自定义规则核销以规则、会员和核销次数排重。
8. 写接口携带 `Idempotency-Key`，数据库同时保存业务请求号。
9. 积分订单、入账/出账批次、核销、余额和预算占用必须事务一致。
10. `xhbs_points_order` 保存处理状态；重复业务请求直接返回原订单结果。

## 19. 验收标准

### 19.1 自动获取

1. 普通会员首次注册自动获得 100 积分，再次登录不重复获得。
2. 个人购买尊享会员卡获得 365，个人购买畅享会员卡获得 188。
3. 续费按对应会员类型获得积分，即使续费卡暂未激活也在支付成功后触发。
4. 政府批量开通会员不获得注册和购卡积分。
5. 借阅 N 册成功后获得 `N × 10`，同一借阅单不重复发放。
6. 购卡和续费积分支付成功后立即可见但冻结，不能用于任何积分消耗。
7. 到达支付时间加 7 天后积分只解冻一次；7 天内退卡只冲正原冻结积分。

### 19.2 消费和会员日

1. 普通日录入实付 100 元图书消费，获得 100 积分。
2. 每月 6、16、26 日录入实付 100 元图书或自营商品，获得 200 积分。
3. 会员日购买储值卡仍按 1 倍计算。
4. 相同门店和小票号不能重复录入。
5. 店员最终积分超过 3000 时整单拒绝，需要由店长重新操作。

### 19.3 消耗和退回

1. 礼品兑换创建兑换单并扣除配置积分。
2. 退书或商品扣回原订单实际发放积分，包括原会员日倍增部分。
3. 会员卡退款只处理原销售单奖励积分，批量采购卡不扣积分。
4. 退款、退货和活动取消重复提交不会重复扣分或退分。

### 19.4 有效期和协议

1. 新入账积分的到期时间为入账时间加一个自然年。
2. 积分消费优先使用最早过期批次。
3. 过期任务重复执行不会重复扣减。
4. 新会员注册时记录最新会员协议版本及同意时间。

### 19.5 预算和权限

1. 积分规则超预算时不能接受新的非承诺操作，并产生预警。
2. 店员单笔最终积分上限确定为 3000，不设置每日累计上限；店长不设业务上限，经理无人工积分权限。
3. 角色额度修改后按生效时间执行并保留历史配置。
4. 员工不能操作数据权限范围外会员。

## 20. 上线前必须确认

以下事项必须在开发对应功能前确定：

1. 1 元 1 积分的小数金额是否统一向下取整；本文按向下取整设计。
2. 人工消费允许补录多少天，以及店员是否只能录入当天消费。
3. 历史积分的协议通知和首个到期日。
4. 积分预计成本的财务换算口径和各年度总预算。
5. 已发布兑换项调整所需积分时，是否只影响新兑换；本文按不影响历史兑换设计。
