# P03 — 积分发放与消耗管理表设计

> 版本：V2.4
>
> 状态：第一期规则目录及数值管理已开发并落测试库，其余业务场景按开发状态逐步接入
>
> 关联文档：[P01-积分体系](P01-积分体系.md)、[P02-积分发放与消耗需求](P02-积分规则活动与任务设计.md)
> 设计原则：不建设活动表和任务表；只新增一张积分管理表，并扩展现有积分订单日志

## 1. 设计结论

第一期数据库调整如下：

| 类型 | 表名 | 处理方式 | 用途 |
|---|---|---|---|
| 积分管理 | `xhbs_points_rule` | 新建，唯一新增业务表 | 配置所有积分获取和消耗类型 |
| 积分日志 | `xhbs_points_order` | 扩展现有表 | 记录每次积分增加、扣减、过期和冲正 |
| 会员余额 | `member` | 增加冻结余额字段 | 分别保存可用积分和冻结积分 |
| 入账批次 | `xhbs_points_user_into_bill_detail` | 增加冻结状态和可用时间 | 保存获得积分、冻结状态及剩余积分 |
| 出账批次 | `xhbs_points_user_out_bill_detail` | 继续使用并修正核销逻辑 | 保存积分消耗 |
| 核销关系 | `xhbs_points_user_bill_detail` | 继续使用并补齐业务写入 | 记录消耗了哪些入账批次 |

不再新增以下表：

- 活动表。
- 任务表。
- 活动参与表。
- 活动核销表。
- 活动日志表。
- `points_event` 技术任务表。

线下活动和付费活动不建立独立活动模型，而是在 `xhbs_points_rule` 中各创建一条有名称、有效期、适用门店和积分额度的自定义规则。

## 2. 两种状态必须区分

需求中的“已有、暂无、开发中”属于项目交付进度，不是业务数据状态：

| 交付状态 | 说明 |
|---|---|
| 已有 | 基础业务已经存在，但仍需核对是否已完整接入积分账务 |
| 暂无 | 当前没有对应业务能力 |
| 开发中 | 已进入需求或开发阶段 |

为支持后台按交付进度查询，规则表增加 `implementation_status` 保存该状态；它只用于展示和筛选，不能代替规则运行状态，也不能据此触发积分业务。

数据库中的状态分别表示：

- 规则状态：草稿、启用、停用、结束。
- 日志状态：待处理、处理中、成功、失败、已冲正。

## 3. 积分管理表 `xhbs_points_rule`

后台页面名称可以使用“积分发放管理”，但表同时配置增加和扣减积分，因此使用 `direction` 区分方向。

一条规则只代表一种积分处理方式。例如“注册会员送 100 分”是一条规则，“暑期读书会核销送 50 分”也是一条规则。

### 3.2 业务场景编码

首期固定场景：

| `scene_code` | 场景 |
|---|---|
| `REGISTER_MEMBER` | 注册普通会员 |
| `BUY_PREMIUM_CARD` | 购买尊享会员 |
| `BUY_ENJOY_CARD` | 购买畅享会员 |
| `RENEW_PREMIUM_CARD` | 续费尊享会员 |
| `RENEW_ENJOY_CARD` | 续费畅享会员 |
| `OFFLINE_EVENT` | 线下活动核销 |
| `BORROW_BOOK` | 借阅图书 |
| `PURCHASE_BOOK` | 购买书籍 |
| `PURCHASE_SELF_GOODS` | 购买店内自营商品 |
| `PURCHASE_STORED_VALUE` | 购买储值卡 |
| `PAID_EVENT` | 付费活动积分抵扣 |
| `GIFT_EXCHANGE` | 兑换礼品 |

| `REFUND_ENJOY_CARD` | 畅享会员卡退订 |
| `RETURN_ORDER` | 退书或商品 |

### 3.3 计算方式

| `calculation_mode` | 计算逻辑 | 适用场景 |
|---|---|---|
| `FIXED` | 固定积分 | 注册、购卡、续费、活动核销、退卡 |
| `PER_ITEM` | 数量 × 每件积分 | 借阅图书 |
| `PER_YUAN` | 实付金额 × 每元积分 | 购书、自营商品、储值卡 |
| `MANUAL` | 有权限员工输入积分 | 礼品兑换等自定义场景 |
| `ORIGINAL_ORDER` | 按原积分订单实际积分处理 | 退书、退商品、退款和冲正 |

规则：

- 所有配置积分保存正数，最终增加或扣减由 `direction` 决定。
- `FIXED` 必须填写大于 0 的 `fixed_points`。
- `PER_ITEM/PER_YUAN` 必须填写大于 0 的 `points_per_unit`。
- `MANUAL` 必须设置 `manual_points_editable = 1`，同时配置单笔上限和操作权限。
- `ORIGINAL_ORDER` 不允许员工填写积分，必须从原积分订单读取。
- `freeze_days > 0` 只允许用于增加积分规则；冻结期从业务支付成功时间开始计算。

### 3.4 规则状态

| 状态 | 说明 |
|---|---|
| `DRAFT` | 草稿，不参与积分计算 |
| `ENABLED` | 已启用且在有效期内可使用 |
| `DISABLED` | 已停用，不接受新业务 |
| `ENDED` | 已结束，保留历史查询 |

启用规则时必须记录 `operator_user_id`、`operator_name` 和 `operation_time`。

固定系统规则启用后不得物理删除。第一期允许有权限的管理员直接修改 `fixed_points` 或 `points_per_unit`；修改只影响后续新订单，历史积分日志使用原规则快照，不受新值影响。`ORIGINAL_ORDER` 规则没有固定积分，仍必须按原积分订单实际值冲正。

### 3.5 索引和约束

- 唯一索引：`uk_points_rule_code(rule_code)`。
- 查询索引：`idx_points_rule_scene(scene_code, direction, status)`。
- 生效索引：`idx_points_rule_effective(status, effective_from, effective_to)`。
- 操作索引：`idx_points_rule_operator(operator_user_id, operation_time)`。
- 同一自动业务场景、门店范围和重叠有效期只能启用一条规则。
- 临时线下活动和付费活动允许多条规则，但 `rule_code`、名称、有效期和适用门店必须明确。

### 3.6 不单建活动表时如何配置活动

例如“2026 暑期读书会核销送 50 分”，直接创建一条规则：

```text
rule_code          = OFFLINE_EVENT_2026_SUMMER_READING
rule_name          = 2026暑期读书会核销奖励
scene_code         = OFFLINE_EVENT
rule_source        = CUSTOM
direction          = ADD
trigger_mode       = AUTO
trigger_event      = MEMBER_CODE_VERIFIED
calculation_mode   = FIXED
fixed_points       = 50
effective_from/to  = 活动开始和结束时间
dept_ids           = 允许核销的门店
member_limit       = 1
```

员工选择该规则并扫描会员动态码。动态码、门店、有效期和次数校验成功后自动发放积分。

“付费活动扣 100 分”同样创建一条 `direction = DEDUCT`、`scene_code = PAID_EVENT` 的自定义规则。

该方案不管理活动报名、场次、宣传页面和人员名单；如果后续需要这些能力，再独立建设活动业务，不影响积分规则表。

### 3.7 规则配置修改日志

第一期不新增规则变更日志表：

- `xhbs_points_rule.operator_user_id/operator_name/operation_time` 保存最近操作人和时间。
- 新建、修改、启用、停用、结束和复制规则时，继续使用现有 `sys_oper_log` 记录请求、操作人、操作时间和结果状态。
- 后台开放规则新增、查看、修改和逻辑删除；规则编码创建后不可修改，其他配置修改只影响后续订单。每次变更由 `sys_oper_log` 留痕，并更新规则最近操作人和操作时间。
- 每一笔积分订单保存规则名称和计算参数快照，保证规则后续停用或复制后历史积分仍可解释。

### 4.1 获取积分

| `RENEW_PREMIUM_CARD` | 已有 | 自动 | 固定 | 365 | 续费支付成功后入账可见，冻结 7 天；批量采购不发 |
| `RENEW_ENJOY_CARD` | 已有 | 自动 | 固定 | 188 | 续费支付成功后入账可见，冻结 7 天；批量采购不发 |
| `OFFLINE_EVENT_TEMPLATE` | 暂无 | 扫码后自动 | 固定 | 待配置 | 线下活动规则模板，后续具体活动可复制为自定义规则 |
| `BORROW_BOOK` | 已开发，待迁移部署 | 自动 | 每册 | 10 | 一次借 N 本获得 `N × 10` |

### 4.2 消耗积分

| 规则编码 | 需求进度 | 方式 | 计算 | 初始值 | 关键规则 |
|---|---|---|---|---:|---|
| `PAID_EVENT_TEMPLATE` | 暂无 | 扫码后自动 | 固定 | 待配置 | 付费活动规则模板，后续具体活动可复制为自定义规则 |
| `GIFT_EXCHANGE` | 暂无 | 手动 | 自定义 | 自定义 | 扫会员码、填写礼品和凭证 |
| `REFUND_PREMIUM_CARD` | 已有 | 自动 | 固定/原单 | 365 | 关联原购卡积分订单，重复退款不重复扣 |
| `REFUND_ENJOY_CARD` | 已有 | 自动 | 固定/原单 | 188 | 关联原购卡积分订单，重复退款不重复扣 |
| `RETURN_ORDER` | 开发中 | 手动触发 | 原单 | 原单实际积分 | 退货员工发起，系统按原单计算 |

“已有”只代表会员注册、购卡或退卡等基础业务存在，不表示对应业务都已改造为从规则表读取。`implementation_status` 与 `status` 相互独立；只有已经接入统一积分服务的场景，才会在业务触发时实际查询并执行该规则。

## 5. 会员日计算

购买书籍和店内自营商品的规则配置：

```text
calculation_mode      = PER_YUAN
points_per_unit       = 1
rounding_mode         = FLOOR
member_day_enabled    = 1
member_day_days       = [6,16,26]
member_day_multiplier = 2.000
```

计算示例：

```text
普通日：floor(实付金额 × 1)
会员日：floor(实付金额 × 1) × 2
```

实付 100 元，普通日获得 100 分，会员日最终获得 200 分。

购买储值卡的 `member_day_enabled = 0`，每月 6/16/26 仍按 1 倍积分。

积分日志必须保存实付金额、基础积分、会员日倍数和最终积分，退货时按原日志实际发放积分扣回，不能按退货当天规则重新计算。

## 6. 修改现有积分日志表 `xhbs_points_order`

### 6.1 当前问题

现有表已经保存订单号、会员、积分数量、前后余额和设备，但存在以下不足：

- 操作人只拼在 `custom_args` 字符串中，不能可靠查询。
- 没有操作人 ID、操作人类型和明确的操作时间。
- 没有处理状态，无法区分待处理、成功和失败。
- 没有规则 ID、规则编码和规则快照。
- 没有结构化业务类型、原业务单号和幂等键。
- `type` 表示虚拟/实体，不能作为增加/扣减方向。
- 扣减订单代码使用 `OUT`，数据库约定为 `OT`，需要统一。

```text
扣减积分：amount < 0
```

`orgin_points` 拼写错误暂时保留，避免影响现有代码；新代码仍需正确映射。后续如要改名，应通过兼容迁移单独处理。

### 6.3 操作人、操作时间和状态

操作人规则：

| 场景 | `operator_type` | `operator_user_id` | `operator_name` |
|---|---|---|---|
| 注册、购卡、续费、借阅自动积分 | `SYSTEM` | null | `SYSTEM` |
| 员工扫码、消费录入、礼品兑换 | `STAFF` | 当前 `sys_user.user_id` | 当前员工姓名快照 |
| Admin 补处理或冲正 | `ADMIN` | 当前 `sys_user.user_id` | 当前管理员姓名快照 |

`operation_time` 保存业务实际发生或员工确认时间，不用 `updated_at` 代替。购卡积分的操作时间为支付成功时间，`available_at = operation_time + 7 天`。

日志状态：

| 状态 | 说明 |
|---|---|
| `PENDING` | 已接收，等待处理 |
| `PROCESSING` | 正在处理积分账务 |
| `SUCCESS` | 积分订单、批次、核销和余额全部成功 |
| `FAILED` | 处理失败，未形成有效积分变动 |
| `REVERSED` | 原成功订单已通过反向订单完整冲正 |

状态变化、操作人和操作时间必须由后端生成，前端不能直接指定。

### 6.4 索引和约束

- 唯一索引：现有 `uniq_order_number(order_number)` 保留。
- 唯一索引：`uk_points_order_business_key(business_key)`。
- 唯一索引：`uk_points_order_idempotency(idempotency_key)`。
- 规则查询：`idx_points_order_rule(rule_id, order_status, operation_time)`。
- 会员查询：`idx_points_order_member(member_id, order_status, operation_time)`。
- 操作人查询：`idx_points_order_operator(operator_user_id, operation_time)`。
- 业务单查询：`idx_points_order_business(business_type, business_order_no)`。
- 原单查询：`idx_points_order_original(original_order_no)`。
- 解冻日志查询：`idx_points_order_availability(availability_status, available_at)`。
- `SUCCESS` 时 `amount != 0`、`completed_time` 和 `operation_time` 必须有效。
- `MANUAL` 时必须存在真实操作人；不允许用 `SYSTEM` 代替员工。
- `REVERSE` 必须填写 `original_order_no`。

### 6.5 订单号前缀

统一使用：

| 前缀 | 说明 |
|---|---|
| `IN` | 增加积分 |
| `OT` | 消耗积分 |
| `EX` | 积分过期 |
| `BK` | 原积分操作的反向处理 |

现有代码生成的 `OUT` 改为 `OT`。历史 `OUT` 单号保留，不执行重命名。

### 6.6 历史日志迁移

`xhbs_points_order` 已有历史数据，新增字段不能一步设置为严格非空。迁移顺序：

1. 先以可空或兼容默认值增加字段和普通索引。
2. 根据 `amount` 回填 `direction`：正数为 `ADD`，负数为 `DEDUCT`。
3. 历史成功数据回填 `order_status = SUCCESS`。
4. `rule_code/scene_code` 回填 `LEGACY`，`rule_name` 回填 `历史积分`。
5. `operator_type/operator_name` 回填 `SYSTEM/LEGACY_SYSTEM`；若 `custom_args` 能可靠解析员工名称，可补充名称但不能猜测员工 ID。
6. `operation_time` 优先取有效的 `completed_time`，否则取 `order_time` 或 `created_at`。
7. `business_key` 回填 `LEGACY:{order_number}`，`idempotency_key` 回填同值。
8. 完成数据核对后再增加唯一索引和非空约束。

历史数据无法确定具体规则或操作人的，必须明确标为 `LEGACY`，不能伪造业务来源。

### 6.7 冻结积分数据结构

#### `member` 增加字段

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `frozen_points` | int | 0 | 已入账可见但当前不可使用的积分 |

余额口径：

```text
可用积分 = member.current_points
冻结积分 = member.frozen_points
会员可见总积分 = current_points + frozen_points
```

所有兑换、活动抵扣、退货扣分和人工扣分只能校验并扣减 `current_points`，不得使用 `frozen_points`。

#### `xhbs_points_user_into_bill_detail` 增加字段

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `availability_status` | varchar(20) | `AVAILABLE` | `FROZEN/AVAILABLE/REVERSED` |
| `available_at` | datetime | null | 计划解冻时间 |
| `unfrozen_at` | datetime | null | 实际解冻时间 |
| `related_biz_type` | varchar(50) | null | 如 `MEMBER_CARD_ORDER` |
| `related_biz_no` | varchar(64) | null | 会员卡销售订单号 |

冻结状态和原有 `bill_status` 分开：

- `availability_status` 管理能否使用。
- `bill_status` 管理可用后是否未核销、部分核销或完全核销。

增加解冻扫描索引：

```text
idx_points_into_unfreeze(availability_status, available_at, id)
```

`member.current_points` 和 `member.frozen_points` 都不得小于 0。

可用余额和冻结余额必须满足：

```text
member.current_points
= 所有 AVAILABLE、未过期、未冲正入账批次的 remaining_points 合计

member.frozen_points
= 所有 FROZEN、未冲正入账批次的 remaining_points 合计
```

积分订单余额校验调整为：

```text
amount
= (after_points - orgin_points)
 + (after_frozen_points - before_frozen_points)
```

这样购卡入账 365 分时，可用余额不变、冻结余额增加 365，订单金额仍为 `+365`。

自动解冻不创建第二笔积分收入订单，因为积分总额没有变化，只是在冻结余额和可用余额之间转移。系统更新原 `IN` 订单及入账批次的可用状态，并记录 `unfrozen_at`；退款冲正则必须新增 `BK` 订单。

## 7. 幂等键

每个自动或手动场景都必须生成唯一 `business_key`：

| 场景 | 建议格式 |
|---|---|
| 注册会员 | `REGISTER:{memberId}` |
| 购卡/续费 | `{sceneCode}:{memberCardOrderNo}:{memberId}` |
| 借阅 | `BORROW_BOOK:{borrowOrderNo}` |
| 消费录入 | `{sceneCode}:{deptId}:{receiptNo}:{memberId}` |
| 线下核销 | `{ruleId}:VERIFY:{memberId}:{sequence}` |
| 礼品兑换 | `GIFT:{redemptionOrderNo}:{memberId}` |
| 会员卡退订 | `{sceneCode}:{refundOrderNo}:{memberId}` |
| 退书/商品 | `RETURN:{returnOrderNo}:{memberId}` |

相同业务键重复提交时返回原积分订单，不重复增加或扣减积分。

## 8. 积分日志处理流程

为了保留失败状态且保持积分账务一致，建议：

1. 接收请求后生成订单号、业务键和幂等键，创建 `PENDING` 日志。
2. 将状态更新为 `PROCESSING`。
3. 开启积分事务并锁定会员行。
4. 校验规则状态、有效期、门店、会员类型、权限、次数和预算。
5. 增加积分时写入积分订单和入账批次；普通积分更新可用余额，冻结规则更新冻结余额。
6. 扣减积分时写入积分订单、出账批次、入账批次核销关系并更新余额。
7. 同事务更新规则 `used_count/used_points` 和订单 `SUCCESS` 状态。
8. 事务失败后在独立错误处理事务中把订单标记为 `FAILED`，保存错误码和错误信息。
9. 定时任务扫描长时间停留在 `PENDING/PROCESSING` 的记录并对账或重试。

不能只写日志和余额而不写入账/出账批次，也不能因为写失败日志而提交一半积分账务。

## 9. 具体业务规则

### 9.1 注册、购卡和续费

- 注册积分以会员 ID 排重，只发一次。
- 购卡和续费以会员卡销售订单号排重。
- 购卡支付成功即触发，即使会员卡排队待生效也不重复发放。
- 尊享、畅享购卡和续费积分统一冻结 7 天，支付成功后立即入账可见但不可使用。
- 冻结期按支付成功时间精确计算：`available_at = pay_time + 7 × 24 小时`。
- 7 天内退卡直接冲正原冻结积分，不扣会员其他可用积分，因此不会产生余额不足。
- 政府或单位批量采购使用明确的 `is_bulk_purchase` 字段判断；对应规则设置 `exclude_bulk_purchase = 1`。
- 不能再通过备注中是否存在单位名称判断。

购卡入账事务：

1. 锁定会员和会员卡销售订单。
2. 创建 `IN` 积分订单，`availability_status = FROZEN`。
3. 创建冻结入账批次，写入 `available_at` 和销售订单号。
4. `member.frozen_points += 365/188`，`member.current_points` 不变。
5. 订单、批次、冻结余额和规则预算同事务提交。

自动解冻：

1. Quartz 解冻任务建议每分钟执行，扫描 `availability_status = FROZEN AND available_at <= 当前时间` 的批次。
2. 按固定顺序锁定会员卡销售订单、积分入账批次和会员行。
3. 再次确认退款单不处于处理中或成功状态，并且积分批次仍为冻结状态。
4. `member.frozen_points -= remaining_points`。
5. `member.current_points += remaining_points`。
6. 批次和原积分订单改为 `AVAILABLE`，写入 `unfrozen_at`。
7. 解冻失败保留冻结状态并进入重试；条件更新保证重复任务不会重复解冻。

正常情况下积分在 `available_at` 后一分钟内变为可用；任务延迟时以 `available_at` 为资格判断时间，不重新顺延 7 天。

退卡冲正：

1. 退卡只允许 `当前时间 < available_at`；自动解冻条件为 `当前时间 >= available_at`，时间边界不重叠。
2. 退款事务锁定顺序与解冻任务一致，避免退款和解冻并发。
3. 退款受理后先把退款单置为处理中；解冻任务看到处理中状态时跳过该批次。
4. 创建关联原 `IN` 订单的 `BK` 反向订单，金额为 `-365/-188`。
5. `member.frozen_points` 减少对应积分，冻结入账批次和原积分订单标记 `REVERSED`。
6. 原购卡规则释放对应 `used_points/used_count`，退卡规则记录一次成功执行。
7. 退款单、反向积分订单、批次、规则计数和冻结余额必须同事务完成；任一步失败不得把退款标记为最终成功。
8. 退款最终失败且已经超过 `available_at` 时，解除退款处理中状态，由下一次解冻任务转为可用。

积分一年有效期仍从购卡积分入账时间开始计算，7 天冻结期包含在一年有效期内。

### 9.2 借阅图书

- 借阅订单和逐册明细成功后自动触发；积分失败不得重复创建借阅单。
- 借阅模块不读取 `book_info`、book history 或库存。每条明细是一册，图书编号和书名均为员工录入的历史快照。
- 借阅服务调用积分服务时传入 `scene_code=BORROW_BOOK`、`business_order_no=借阅单号`、`base_quantity=逐册明细数`，不传最终积分。
- 积分服务查询生效的 `xhbs_points_rule`，校验 `direction=ADD`、`trigger_mode=AUTO`、有效期、门店范围、预算和上限。
- 初始规则为 `calculation_mode=PER_ITEM`、每册 10 分；最终积分=`base_quantity × 当时规则每册积分`。
- 全局幂等键为 `BORROW_BOOK:{borrowOrderNo}`，同一借阅订单只产生一笔正向积分日志。
- `xhbs_points_order.base_quantity`保存册数，`calculation_snapshot` 保存每册积分、规则版本和计算结果。
- 规则停用或未命中时不影响借阅成功，但保存未发放原因并允许用原业务键重试。
- 不新建借阅积分任务表；复用 `xhbs_points_order` 的处理状态和业务键保存待处理/失败记录，Quartz 只扫描超时记录做重试和对账。
- 只有误录撤销才关联原积分订单冲正；正常归还、损坏归还、遗失和借转购不撤销奖励。

### 9.3 购书、自营商品和储值卡

- 员工先扫描会员动态码，再选择积分规则并录入小票号和实付金额。
- 积分由系统根据实付金额计算，员工不能直接修改最终积分。
- 购书和自营商品在每月 6/16/26 自动使用 2 倍。
- 储值卡始终按 1 倍。
- 相同门店、小票号、会员和业务类型不能重复录入。

### 9.4 线下活动核销和付费活动

- 每个临时活动直接在积分管理表创建一条自定义规则。
- 规则填写名称、方向、固定积分、有效期、门店和单会员次数。
- 员工选择规则并扫描会员动态码后，系统自动增加或扣减积分。
- 该方案不保存报名、场次和参与名单，只保存积分操作日志。

### 9.5 礼品兑换

- 员工扫描会员码，选择“礼品兑换”规则。
- 填写礼品名称、扣减积分、备注和凭证。
- 受岗位权限和单笔积分上限控制。
- 第一期没有礼品目录和库存表，兑换内容保存在日志快照和备注中。

### 9.6 会员卡退订

- 必须关联原会员卡销售订单和原积分订单。
- 政府批量采购没有发过积分，不执行扣分。
- 同一退款订单只能处理一次。
- 7 天内奖励积分仍处于冻结状态，退卡只冲正原冻结批次，不会占用会员其他可用积分。
- 超过 7 天积分自动可用，同时既定 7 天退卡窗口结束，不再进入退卡积分冲正流程。

### 9.7 退书或商品

- 员工发起退货积分处理，但不能手工填写任意扣分数。
- 必须关联原消费积分订单。
- 按原订单实际获得的积分扣回，包括会员日倍增积分。
- 部分退货按原订单明细和原规则快照计算，累计扣回不得超过原订单发放积分。

## 10. 后台页面

### 10.1 系统管理—积分规则管理

入口位于一级菜单“系统管理”下的二级菜单“积分规则管理”，数据来自 `xhbs_points_rule`。

查询条件：规则名称、场景、方向、触发方式、状态、生效时间。

列表字段：

- 规则编码。
- 获取/消耗类型。
- 自动/手动。
- 计算方式和积分值。
- 会员日倍数。
- 生效时间。
- 适用门店。
- 预算、已使用积分。
- 状态。
- 最近操作人。
- 最近操作时间。

操作：新增、查看详情、修改、删除、启用、停用和结束。

- 规则编码创建后不可修改。
- 已有积分流水或 `used_count > 0` 的规则禁止删除，只能停用或结束。

- 固定积分、每件积分和每元积分均可修改，但只作用于后续新订单。
- 未配置积分值的固定或单位规则只能保存为草稿，配置完整后才能启用。
- 单会员次数、规则总次数、积分预算和单笔积分上限填 `0` 或留空均表示“不限”，数据库统一保存为 `NULL`。

列表字段：

- 会员信息。
- 规则名称。
- 获取/消耗方向。
- 积分数量。
- 操作前后余额。

- 操作人。
- 操作时间。
- 状态。
- 失败原因。

### 10.3 会员端积分展示

会员端余额接口增加：

| 字段 | 说明 |

| `totalVisiblePoints` | 可见总积分，即可用积分加冻结积分 |
| `availablePoints` | 当前可消费积分 |

### 10.4 会员列表积分操作

会员列表“积分”弹窗提供“加积分”和“减积分”两个入口，并在弹窗下方展示该会员的积分记录。

- 规则下拉只返回当前启用、处于有效期内、适用于该会员门店和卡类型的人工固定积分规则。
- 加积分只展示 `direction=ADD` 的规则，减积分只展示 `direction=DEDUCT` 的规则。
- 选择规则后自动带出 `fixed_points`，积分数值只读；请求只提交规则 ID 和备注，后端必须重新查询并锁定规则，不能信任前端积分数值。
- 后端在同一事务中写积分订单、入账或出账明细、更新会员余额和规则使用次数；扣减前必须校验可用余额。
- 积分记录展示规则名称、有符号积分、操作前后余额、备注、操作人、状态和操作时间。

接口：

- `GET /member/{memberId}/points/rules?direction=ADD|DEDUCT`：查询可选规则。
- `POST /member/{memberId}/points/adjust`：按规则加减积分，请求体为 `ruleId`、`description`。
- `GET /member/{memberId}/points`：查询会员积分记录。

会员端总积分可以包含冻结积分，但兑换、抵扣和扣分确认页面必须明确使用 `availablePoints`。冻结流水展示“冻结中，预计 YYYY-MM-DD HH:mm 可用”。

## 11. 权限

- 积分规则管理权限：`system:pointsRule:list/query/add/edit/remove`，第一期默认只分配给系统管理员。
- 自动积分不需要员工手工权限，但受规则状态、有效期和预算限制。
- 店员手工积分操作单笔上限确定为 3000，不设置每日累计上限。
- 3000 按本次最终积分绝对值校验，增加和扣减都适用；会员日倍增后再校验。
- 最终积分超过 3000 时整单拒绝，由店长操作，不允许店员拆分同一小票或同一业务单规避。
- 相同小票/业务单通过唯一 `business_key` 排重，即使拆成多次请求也不能重复积分。
- 店长不设业务上限，但仍受规则和预算限制。
- 经理默认无手工积分权限。
- 线下扫码、礼品兑换、消费录入和冲正都要校验后端权限及部门数据范围。
- 第一阶段复用现有角色、菜单权限和 `sys_config` 配置额度，不新增权限策略表；建议配置 `points.manual.staff.max_per_order = 3000`。

## 12. 对账

每日检查：

1. `SUCCESS` 积分订单与会员余额变化是否一致。
2. 增加积分订单是否都有入账批次。
3. 扣减积分订单是否都有出账批次和入账核销关系。
4. `amount = after_points - orgin_points` 是否成立。
5. 规则 `used_count/used_points` 是否与成功且未冲正日志一致。
6. 是否存在重复 `business_key` 或 `idempotency_key`。
7. 手工操作是否都有真实操作人、操作时间和凭证。
8. 冲正订单是否关联原积分订单。
9. `member.frozen_points` 是否等于冻结入账批次余额合计。
10. 是否存在超过可用时间仍未解冻、已经退款却仍冻结或被重复解冻的批次。

## 13. 验收标准

1. 只新增 `xhbs_points_rule` 一张业务表，不创建活动表和任务表。
2. 获取和消耗积分均可在同一页面配置，并通过方向区分。
3. 每条规则有状态、最近操作人和最近操作时间。
4. 每笔积分日志有操作人、操作时间和处理状态。
5. 普通会员首次注册获得 100 分且不会重复发放。
6. 尊享/畅享购卡和续费分别获得 365/188 分，批量采购不发。
7. 借阅 N 本获得 `N × 10`，同一借阅单不重复发放。
8. 购书和自营商品普通日 1 元 1 分，会员日最终 2 倍。
9. 储值卡 1 元 1 分但不参加会员日。
10. 线下核销和付费活动可以通过自定义规则配置固定积分。
11. 礼品兑换可手工扣分并保存员工和凭证。
12. 退卡和退货关联原单，不重复扣分。
13. 并发处理不能突破规则次数或预算。
14. 成功日志可以追溯规则、业务单、操作人、积分批次和余额。
15. 失败日志保留失败原因并可安全重试。
16. 购卡和续费积分支付后立即可见但不可消费，7 天到期后只自动解冻一次。
17. 7 天内退卡冲正原冻结积分，不扣其他可用积分。
18. 店员单笔最终积分超过 3000 时整单拒绝，同一小票不能拆单规避。

## 14. 上线前确认

1. “积分发放管理”页面是否同时展示消耗规则；本文建议用两个页签展示同一张表。
2. 线下活动是否只需要积分核销，不需要报名、场次和参与名单。
3. 付费活动是否在现场扫码时扣分；本文按现场扫码设计。
4. 礼品兑换是否首期不管理礼品目录和库存。
5. 人工消费允许补录多少天。
