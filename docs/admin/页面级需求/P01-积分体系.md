# P01 — 积分体系

> 版本：V1.2
>
> 状态：现状分析及目标需求稿，待业务确认、待开发
>
> 适用范围：Admin 后台、员工端小程序、会员端小程序、ERP 会员导入、积分账户和积分流水

> 新增积分获取、消耗、会员日、线下核销、预算、岗位额度和批量采购规则见 [P02-积分发放与消耗需求](P02-积分规则活动与任务设计.md)，单表方案见 [P03-积分发放与消耗管理表设计](P03-积分发放与消耗管理表设计.md)。P02/P03 是本期实现基线，并替代本文早期“积分事项”类型及有效期未确认的假设。

## 1. 文档目标

本文档基于当前代码和数据库结构，说明系统现有积分能力、已发现的问题及目标积分体系需求。

本期目标不是只完成 `member.current_points` 的加减，而是建立能够追踪以下信息的积分账务闭环：

1. 积分因为什么业务产生。
2. 每一笔积分由谁、在哪个门店、通过什么渠道操作。
3. 当前余额由哪些入账批次组成。
4. 消耗积分时核销了哪些入账批次。
5. 积分过期、退回或冲正时如何恢复原账务关系。
6. 会员余额、积分订单、入账明细、出账明细和核销明细是否一致。

## 2. 术语与账户口径

当前代码和表注释同时使用“积分”和“书城币”。本需求统一对外展示为“积分”，数据库表名暂不改名。

| 概念 | 当前存储 | 目标定义 |
|---|---|---|
| 可用积分 | `member.current_points` | 会员当前可以消费的积分余额 |
| 冻结积分 | `member.frozen_points` | 已入账可见但暂不可消费的积分余额 |
| 可见总积分 | 计算值 | `current_points + frozen_points`，用于会员端总积分展示 |
| 累计获得积分 | `member_ext.total_points` | 会员历史累计获得积分，只随有效入账增加，不因正常消费和过期减少 |
| 等级积分 | `member_ext.level_points` | 用于会员等级计算的独立累计值，是否随事项增加由积分事项配置决定 |
| 积分订单 | `xhbs_points_order` | 每次积分变化的总账记录 |
| 入账批次 | `xhbs_points_user_into_bill_detail` | 每次获得积分形成的可消费批次 |
| 出账记录 | `xhbs_points_user_out_bill_detail` | 每次消费、过期或扣减形成的出账记录 |
| 核销明细 | `xhbs_points_user_bill_detail` | 出账记录与被消耗入账批次之间的对应关系 |

积分只能使用整数，不允许小数，不允许账户余额为负数。

如果未来支持积分抵扣人民币，必须单独配置兑换比例和抵扣上限；当前代码中的积分与金额字段不能直接视为同一单位。

## 4. 当前问题和风险

### 4.1 P0：扣减没有完成入账批次核销

当前扣减积分只执行：

1. 插入积分订单。
2. 插入出账记录。
3. 修改 `member.current_points`。

没有执行：

- 按顺序选择可用入账批次。
- 扣减入账批次的 `remaining_points`。
- 更新入账批次 `bill_status`。
- 写入 `xhbs_points_user_bill_detail`。

结果是余额虽然减少，但无法追踪“扣掉的是哪一批积分”，积分过期和冲正也无法正确实现。

### 4.2 P0：ERP 期初积分没有总账记录

ERP 导入会员时会直接把 Excel“当前积分”写入 `member.current_points`，但不会生成积分订单和入账批次。

因此以下公式无法成立：

```text
当前积分 = 所有未核销入账批次余额之和
```

目标系统必须为导入的期初积分创建“ERP 期初积分”入账订单及入账批次。

### 4.3 P0：积分操作权限和数据范围不足

- Admin 增加积分只有一个通用权限 `member:member:points`，没有区分加分、扣分和冲正。
- 员工端积分接口当前只验证员工身份，没有完整的操作权限校验。
- 员工端列表没有把员工可见部门范围传入积分 SQL。
- 以路径参数传入的会员 ID 没有在现有员工身份拦截器中完成部门范围校验。

目标系统必须对积分列表、详情、增加、扣减、冲正和导出分别授权，并应用部门数据权限。

### 4.4 P0：Admin 后端缺少参数保护

Admin 的积分请求对象没有非空、正整数和最大值校验，Service 也没有统一参数保护。绕过前端后可能传入 `null`、0 或负数。

所有校验必须在 Service 事务入口再次执行，不能只依赖页面控件或 Controller。

### 4.5 P1：积分事项未落库

员工端积分事项由 Controller 写死，并且没有根据 `direction` 返回不同事项，也不校验提交的事项 ID。

目标系统需要独立的积分事项配置表，积分操作必须保存事项快照。

### 4.6 P1：订单号规则不一致

数据库注释和部分文档规定出账前缀为 `OT`，当前代码实际生成 `OUT`。现有文档中同时出现 `OT` 和 `OUT`。

目标统一使用：

| 前缀 | 类型 |
|---|---|
| `IN` | 入账 |
| `OT` | 消费或人工扣减 |
| `EX` | 过期扣减 |
| `BK` | 退回或冲正 |

历史 `OUT` 订单继续兼容查询，但新订单不再生成 `OUT`。

### 4.7 P1：操作人和业务来源不可结构化查询

当前操作人被拼接到 `custom_args`，`operation_device` 又被部分接口当作员工姓名返回，客户端 IP 固定写成 `127.0.0.1`。

目标订单需要独立保存操作人 ID、姓名、门店、渠道、真实客户端 IP、关联业务单号和请求幂等号。

### 4.8 P1：累计积分没有维护

`member_ext.total_points` 只在会员导入时写入，正常增加积分时不会增加，因此“累计积分”无法代表真实历史累计获得积分。

### 4.9 P1：文档与接口返回不一致

现有接口文档声明返回订单号、变动前积分、变动后积分、会员姓名和员工姓名等字段，但部分 Controller 实际只返回成功标记、变动值和时间。

目标接口和文档必须使用统一 DTO，禁止 Controller 临时拼装不同结构。

### 4.10 P1：缺少积分测试

当前没有积分 Service、并发扣减、重复请求、批次核销、过期和冲正相关自动化测试。

## 5. 目标账务闭环

在常规积分账户下，必须始终满足：

```text
member.current_points
= SUM(xhbs_points_user_into_bill_detail.remaining_points)
  WHERE member_id = 当前会员
    AND account_type = 0
    AND is_del = 0
    AND availability_status = 'AVAILABLE'
    AND 批次未失效
```

冻结积分必须满足：

```text
member.frozen_points
= SUM(xhbs_points_user_into_bill_detail.remaining_points)
  WHERE member_id = 当前会员
    AND account_type = 0
    AND is_del = 0
    AND availability_status = 'FROZEN'
```

每一笔积分订单满足：

```text
amount
= (after_points - orgin_points)
 + (after_frozen_points - before_frozen_points)
```

其中入账 `amount > 0`，出账和过期 `amount < 0`。普通入账改变可用余额；购卡冻结入账只改变冻结余额。

每一笔出账满足：

```text
out_bill.points = SUM(points_user_bill_detail.points)
```

每一个入账批次满足：

```text
into_bill.points
= into_bill.remaining_points
 + SUM(points_user_bill_detail.points)
 + 已冲正或失效数量
```

积分订单、批次、核销明细和会员余额必须在同一个数据库事务内提交。

## 6. 积分事项配置

建议新增 `points_reason` 表。

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `reason_code` | 稳定唯一编码，如 `MANUAL_ADD` |
| `reason_name` | 页面展示名称 |
| `direction` | `ADD` 或 `DEDUCT` |
| `default_points` | 默认积分，可空 |
| `min_points` | 单次最小值 |
| `max_points` | 单次最大值 |
| `valid_days` | 本事项产生积分的有效天数；空表示按全局策略 |
| `affect_total_points` | 是否计入累计积分 |
| `affect_level_points` | 是否计入等级积分 |
| `require_related_order` | 是否必须关联业务单号 |
| `require_remark` | 是否必须填写备注 |
| `allowed_channels` | ADMIN、STAFF、SYSTEM、POS 等 |
| `manual_allowed` | 是否允许人工选择 |
| `status` | 启用或停用 |
| `sort` | 排序 |
| 创建人、修改人、时间 | 审计字段 |

建议预置事项：

| 编码 | 名称 | 方向 | 默认方式 |
|---|---|---|---|
| `ERP_OPENING_BALANCE` | ERP 期初积分 | ADD | 仅系统导入可用 |
| `MANUAL_ADD` | 人工增加 | ADD | 备注必填 |
| `ACTIVITY_BONUS` | 活动赠送 | ADD | 可配置默认值 |
| `BORROW_BOOK` | 借阅图书 | ADD | 自动规则，按逐册明细数计算 |
| `PURCHASE_REWARD` | 购书返利 | ADD | 自动业务事项，默认关闭 |
| `MANUAL_DEDUCT` | 人工扣减 | DEDUCT | 备注必填 |
| `GIFT_REDEEM` | 礼品兑换 | DEDUCT | 关联兑换单 |
| `PURCHASE_OFFSET` | 购书抵扣 | DEDUCT | 关联购书单 |
| `POINTS_EXPIRED` | 积分过期 | DEDUCT | 仅系统任务可用 |
| `REVERSAL` | 原单冲正 | 按原单决定 | 仅冲正流程可用 |

积分操作时必须在订单中保存事项编码和名称快照。以后修改事项名称不能改写历史记录。

## 7. 增加积分流程

### 7.1 输入

| 字段 | 必填 | 规则 |
|---|---|---|
| 会员 ID | 是 | 会员必须存在且状态正常 |
| 事项 ID | 是 | 必须为启用的增加事项 |
| 积分值 | 是 | 正整数，符合事项最小值和最大值 |
| 关联业务类型 | 条件必填 | 自动奖励或事项要求时必填 |
| 关联业务单号 | 条件必填 | 用于防止同一业务重复奖励 |
| 备注 | 条件必填 | 人工调整时必填 |
| 幂等键 | 写接口必填 | 防止重复提交 |

### 7.2 事务步骤

1. 校验会员、权限、数据范围、事项、积分范围和关联业务。
2. 校验幂等键及“事项 + 关联业务单号”的业务唯一性。
3. `SELECT member ... FOR UPDATE` 锁定会员。
4. 读取操作前余额并计算操作后余额，检查整数上限。
5. 创建 `IN` 积分订单。
6. 创建入账批次，`points=remaining_points=本次积分`。
7. 根据事项有效期写入 `expired_time`。
8. 更新 `member.current_points`。
9. 按事项配置更新 `member_ext.total_points` 和 `level_points`。
10. 写操作日志并提交事务。

### 7.3 返回

```text
pointsRecordId
beforePoints
pointsDelta
afterPoints
reasonCode
reasonName
expiredAt
operatedAt
```

## 8. 消耗积分流程

### 8.1 核销顺序

积分消耗采用“先过期先使用，同过期时间先进先出”：

1. 有过期时间的批次优先于永久有效批次。
2. 过期时间较早的批次优先。
3. 过期时间相同时，创建时间较早的批次优先。
4. 已过期、已删除、余额为 0 的批次不能使用。

### 8.2 事务步骤

1. 校验会员、权限、数据范围、扣减事项、积分值、备注和关联业务。
2. 校验幂等键和关联业务唯一性。
3. 锁定会员行并校验 `current_points >= 本次扣减积分`。
4. `FOR UPDATE` 查询可用入账批次。
5. 再次校验可用批次余额总和足够。
6. 创建 `OT` 积分订单，订单 `amount` 保存负数。
7. 创建出账记录，`points` 保存正数。
8. 按顺序扣减各入账批次 `remaining_points`。
9. 每使用一个入账批次，插入一条 `xhbs_points_user_bill_detail`。
10. 入账批次余额为 0 时状态改为完全核销，否则改为核销中。
11. 出账全部分配后状态改为完全核销。
12. 更新 `member.current_points` 并提交事务。

`xhbs_points_user_out_bill_detail.remaining_points` 统一表示“该出账记录尚未分配核销的积分”，初始为本次扣减积分，核销完成后必须为 0。不能再存会员操作后的全局余额。

## 9. ERP 期初积分

新增会员从 ERP Excel 导入时：

1. 读取“当前积分”和“总积分”。
2. 会员当前积分大于 0 时，创建 `IN` 期初积分订单。
3. 创建等额入账批次，事项为 `ERP_OPENING_BALANCE`。
4. 订单操作前余额为 0，操作后余额为 Excel 当前积分。
5. `member.current_points` 等于该入账批次余额。
6. Excel 总积分保存至 `member_ext.total_points`，不得用当前积分覆盖总积分。
7. 保存导入批次号、原始会员编号和 ERP 来源信息。

对历史已导入会员应提供一次性数据修复脚本：

- 如果当前积分大于 0 且不存在可用入账批次，补建期初积分订单和批次。
- 修复前输出差异清单，不能静默修改生产数据。

## 10. 积分过期

系统按入账批次执行积分过期，有效期确定为入账时间起一个自然年。

### 10.1 推荐策略

- 所有新入账积分默认有效期为一个自然年。
- 入账批次写入明确的 `expired_time`。
- 过期精确到日，任务在次日凌晨处理前一日到期批次。
- 只处理 `remaining_points > 0` 的到期批次。

### 10.2 过期处理

1. 定时任务扫描到期且仍有余额的入账批次。
2. 按会员分组并锁定会员。
3. 创建 `EX` 订单和出账记录。
4. 使用核销明细关联到具体到期入账批次。
5. 将批次剩余积分扣为 0，并减少 `member.current_points`。
6. 写处理批次、任务日志和失败重试记录。
7. 重复执行任务不能重复过期同一批积分。

如会员开启积分变动提醒，可在过期前发送提醒；提醒失败不能回滚积分账务。

## 11. 冲正与退回

积分流水不能直接编辑或删除。错误积分必须通过冲正生成反向订单。

### 11.1 冲正增加积分

- 只能冲正原入账批次仍未消费的部分。
- 如果原积分已被消费，默认禁止冲正并提示先处理关联业务。
- 冲正后减少当前积分；是否减少累计积分和等级积分按原事项配置反向处理。

### 11.2 冲正扣减积分

- 根据原订单的核销明细，把积分退回原入账批次。
- 恢复各批次 `remaining_points` 和状态。
- 已经过期的原批次退回后不能重新成为可用积分；应按业务确认选择退回新批次或拒绝冲正。

### 11.3 公共规则

- 生成 `BK` 订单并关联原订单号。
- 同一原订单只能成功冲正一次，部分冲正必须有累计上限。
- 备注和冲正原因必填。
- 仅有专门权限的管理员可以操作。
- 原订单和冲正订单都必须保留，禁止逻辑删除替代冲正。

## 12. 自动积分与其他业务联动

借阅业务已通过统一积分服务调用 `BORROW_BOOK` 规则（待数据库迁移和部署）；购书业务尚未接入。业务模块禁止直接修改 `member.current_points`。

### 12.1 借阅奖励

- 默认关闭，业务确认奖励规则后启用。
- 可配置按借阅单、按册数或固定积分奖励。
- 关联借阅单号，并建立唯一约束，防止重复奖励。
- 借阅取消时通过冲正处理，不直接删除流水。

### 12.2 购书返利

- 默认关闭。
- 可配置按实付金额比例或固定值计算。
- 只对已付款订单发放。
- 退款时按原积分订单冲正。

### 12.3 积分抵扣购书

- 购书单中的 `points_deduct` 必须关联一笔真实 `OT` 积分订单。
- 积分扣减和购书单付款状态应在同一业务事务内闭环，或通过可靠事件和补偿机制闭环。
- 购书取消或退款时生成 `BK` 退回积分。
- 在兑换比例、抵扣上限和退款规则未确认前，不开放积分抵扣入口。

## 13. 数据模型调整建议

### 13.1 `xhbs_points_order`

建议增加：

| 字段 | 说明 |
|---|---|
| `change_type` | `IN`、`OT`、`EX`、`BK`，禁止只通过订单号推断 |
| `reason_id` | 积分事项 ID |
| `reason_code` | 积分事项编码快照 |
| `reason_name` | 积分事项名称快照 |
| `related_biz_type` | 借阅单、购书单、活动、ERP 导入等 |
| `related_order_no` | 关联业务单号 |
| `original_order_number` | 冲正时关联原积分订单 |
| `operator_id` | 操作人 ID |
| `operator_name` | 操作人姓名快照 |
| `dept_id` | 操作门店或部门 |
| `channel` | ADMIN、STAFF、SYSTEM、POS、ERP_IMPORT |
| `request_no` | 幂等或业务请求号 |
| `order_status` | 成功、已冲正、部分冲正等 |
| `availability_status` | 冻结、可用、已冲正或不适用 |
| `available_at/unfrozen_at` | 计划可用时间和实际解冻时间 |
| `before_frozen_points/after_frozen_points` | 操作前后冻结余额 |
| `remark` | 操作备注，与事项名称分开 |

建议为 `related_biz_type + related_order_no + reason_code` 和 `request_no` 建立适当唯一约束。

现有拼写错误字段 `orgin_points` 暂时兼容，Java DTO 和新文档统一使用 `beforePoints`；如迁移改名需同步所有 SQL 和历史数据。

### 13.2 入账批次

现有字段基本可复用，需要补充或明确：

- `order_no_src` 唯一关联入账积分订单。
- `remaining_points` 必须由核销事务维护。
- 增加 `availability_status`，区分 `FROZEN/AVAILABLE/REVERSED`。
- 增加 `available_at` 和 `unfrozen_at`；购卡和续费积分支付成功后冻结 7 天。
- `expired_time` 使用可空 DATETIME，不能使用无效零日期。
- 增加适合“会员 + 状态 + 过期时间 + 创建时间”的核销索引。
- 可用状态与核销状态分开：前者管理冻结/可用/冲正，后者管理未核销/部分核销/完全核销。

### 13.3 出账和核销明细

- 出账 Mapper 必须返回主键 ID。
- 补齐 `xhbs_points_user_bill_detail` 领域对象、Mapper 和 Service。
- 核销明细增加原积分订单号或通过入账、出账 ID 可完整追溯。
- 对同一入账批次、出账记录的重复分配增加约束或事务校验。

### 13.4 余额和累计积分

- `member.current_points` 是可用积分缓存余额。
- `member.frozen_points` 是冻结积分缓存余额。
- 会员端可见总积分为两者之和，并分别展示“可用”和“冻结”。
- `member_ext.total_points` 是累计获得值。
- `member_ext.level_points` 是否变化由事项配置决定。
- 任何积分写操作均不能通过通用会员编辑接口直接修改这三个字段。

积分记录页展示：

- 当前可用积分。
- 本年度获得积分。
- 即将过期积分及最近过期时间（启用过期规则后）。

会员只能查看自己的积分订单。逻辑删除、失败或初始化状态订单不能对会员展示。

建议权限：

| 权限标识 | 用途 |
|---|---|
| `points:record:list` | 查询积分流水 |
| `points:record:query` | 查看积分详情 |
| `points:record:export` | 导出积分流水 |
| `points:adjust:add` | 人工增加积分 |
| `points:adjust:deduct` | 人工扣减积分 |
| `points:adjust:reverse` | 冲正积分订单 |
| `points:reason:list` | 查询积分事项 |
| `points:reason:add` | 新增积分事项 |
| `points:reason:edit` | 修改、启停积分事项 |
| `points:expire:execute` | 手工补执行过期任务，仅管理员 |

数据范围规则：

1. 超级管理员可查看全部会员积分。
2. 普通员工只能操作其部门数据范围内的会员。
3. 列表 SQL 必须直接应用数据范围，不能查询全量后在内存过滤。
4. 详情和写接口都要根据目标会员再次校验数据范围。
5. 导出遵循与列表完全一致的数据范围和查询条件。

## 18. 幂等、并发和安全

1. 所有积分写接口必须携带 `Idempotency-Key` 或等价业务请求号。
2. 幂等键应进入数据库订单，不能只依赖 Redis 五分钟缓存。
3. 同一会员积分写操作使用会员行锁串行化。
4. 同一入账批次核销时必须加行锁。
5. 订单号使用数据库唯一索引并增加冲突重试，不能只依赖随机数概率。
6. 单次积分上限由事项配置和角色权限共同控制。
7. 高额人工调整可增加复核审批，本期至少预留审批状态字段。
8. 积分余额及流水不能通过会员通用编辑、Excel更新或直接删除修改。
9. 操作日志不得记录令牌等敏感信息。

## 19. 对账与异常修复

建议提供只读对账任务和 Admin 异常列表。

检查项：

1. `member.current_points` 是否等于可用且有效入账批次剩余积分合计。
2. `member.frozen_points` 是否等于冻结且未冲正入账批次剩余积分合计。
3. 每笔订单是否满足可用余额变化加冻结余额变化等于订单积分。
4. 每笔出账是否已完全分配到核销明细。
5. 入账批次初始积分是否等于剩余积分、核销积分和失效积分之和。
6. 是否存在负余额、负批次余额、跨会员核销。
7. 是否存在重复关联业务奖励或扣减。
8. 是否存在有余额但已经过期的批次。
9. 是否存在超过可用时间仍未解冻、已退款仍冻结或重复解冻的购卡积分。

异常修复不能直接改余额，应生成修复方案和审计记录，经确认后执行补单或冲正。

## 21. 验收标准

### 21.1 增加积分

1. 会员余额 100，增加 50 后余额为 150。
2. 生成一条 `IN` 订单和一条余额为 50 的入账批次。
3. 订单保存事项、操作人、门店、渠道和关联业务。
4. 重复提交相同幂等键只成功一次。
5. 同一业务单号重复奖励被拒绝。

### 21.2 扣减积分

1. 会员存在 30、50 两个入账批次，扣减 60 后，第一批为 0、第二批为 20。
2. 生成一条 `OT` 订单、一条出账记录和两条核销明细。
3. 会员余额、入账批次剩余合计均减少 60。
4. 余额不足、批次余额不足、积分为 0 或负数时全部拒绝且不产生部分数据。

### 21.3 并发与事务

1. 两名员工同时扣减同一会员积分，不得超扣或产生负余额。
2. 写订单、批次、核销明细或余额任一步失败，整笔事务回滚。
3. 同时增加和扣减时，操作前后余额必须连续且无丢失更新。

### 21.4 ERP 期初积分

1. 导入当前积分 100 的新会员后，会员余额和期初入账批次均为 100。
2. 期初积分可以参与后续正常核销。
3. Excel 总积分和当前积分分别保存，不互相覆盖。

### 21.5 权限与数据范围

1. 无加分权限员工不能调用增加接口。
2. 无扣减权限员工不能调用扣减接口。
3. 员工不能查看或调整数据范围外会员的积分。
4. 会员只能查看自己的积分记录。
5. 导出结果与页面筛选和数据权限一致。

### 21.6 过期和冲正

1. 过期任务重复执行不会重复扣积分。
2. 只扣除批次尚未使用的积分。
3. 扣减冲正能够按原核销关系恢复批次。
4. 已冲正订单不能再次全额冲正。

## 22. 待业务确认事项

以下剩余规则实现相关功能前仍需确认：

1. 积分和人民币的兑换比例、单笔抵扣上限及退款规则。
2. `member_ext.total_points` 是否包含 ERP 历史累计积分和线下核销赠送积分。
3. `level_points` 的获取、扣回和等级升降规则。
4. 注销、过期或停用会员是否允许人工调整积分。建议默认禁止，仅管理员冲正例外。
5. 是否允许跨门店员工调整积分。建议默认遵循部门数据权限。
6. 高额积分调整是否需要双人复核及复核阈值。
7. 历史积分的协议通知方式和首个到期日。

在相关规则和 P02 分阶段能力上线前，现有未闭环的自动奖励、积分抵现和高额人工调整入口保持关闭。

实现目标积分体系时应同时交付：

7. 对账任务、购卡积分解冻任务、过期任务和运行日志。
8. 接口文档、数据结构、权限设计、业务流程和测试报告同步更新。
