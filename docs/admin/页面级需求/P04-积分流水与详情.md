# P04 — 积分流水与详情

> 版本：V1.0
>
> 状态：一期设计稿，待开发
>
> 适用范围：Admin 管理后台、积分账务服务、积分到期任务
>
> 关联文档：[P01-积分体系](P01-积分体系.md)、[P03-积分发放与消耗管理表设计](P03-积分发放与消耗管理表设计.md)

## 1. 设计结论

一期只新增一个 Admin 二级菜单页面 **积分流水**，页面内提供：

1. 积分流水分页列表；
2. 积分详情弹窗或抽屉；
3. 详情内展示入账批次、出账记录和批次核销关系。

一期不单独建设入账批次、出账记录、核销分摊和过期任务页面，也不新增 `xhbs_points_batch_allocation` 表。

账务数据继续使用现有四张表：

| 表 | 一期职责 |
|---|---|
| `xhbs_points_order` | 积分业务总流水，作为列表主表 |
| `xhbs_points_user_into_bill_detail` | 生效积分入账批次、剩余积分和到期时间 |
| `xhbs_points_user_out_bill_detail` | 消费、人工扣减、过期和冲销出账记录 |
| `xhbs_points_user_bill_detail` | 出账记录与入账批次之间的核销分摊关系 |

`xhbs_points_user_bill_detail` 已包含 `into_bill_id`、`out_bill_id` 和 `points`，其职责等同于批次分摊表，因此不得重复创建同类表。

## 2. 一期范围

### 2.1 本期实现

- 全局积分流水分页查询；
- 按会员、积分规则、方向、操作类型、状态和时间筛选；
- 查看单笔积分流水详情；
- 查看增加积分形成的入账批次；
- 查看扣减、过期或冲销形成的出账记录；
- 查看一次出账实际核销了哪些入账批次；
- 所有积分消费统一按生效批次先进先出核销；
- 未消费积分按命中规则配置的有效期过期，默认自生效之日起满 360 天；
- 入账批次已消费一部分时，只过期 `remaining_points`；
- 积分消费、过期、冲销和会员余额在同一事务内更新；
- 管理端列表和详情应用部门数据权限。

### 2.2 本期不实现

- 独立的入账批次菜单；
- 独立的出账单菜单；
- 独立的核销分摊菜单；
- 人工修改入账批次余额；
- 人工修改或删除积分流水；
- 积分过期任务的可视化执行记录页面；
- 积分对账异常处理页面；
- 新建活动表、任务表或批次分摊表。

### 4.4 列表主查询

列表以 `xhbs_points_order` 为主表：

```sql
SELECT p.*, m.name AS member_name, m.phone AS member_phone,
       m.card_no AS member_no, m.dept_id
FROM xhbs_points_order p
JOIN member m ON m.id = p.member_id
WHERE p.is_del = 0
ORDER BY p.operation_time DESC, p.id DESC;
```

部门数据权限通过会员的 `m.dept_id` 应用，不允许只在前端过滤。

规则名称、操作人和业务信息必须使用积分订单中的快照字段，避免规则改名后历史流水展示变化。

## 5. 积分详情

### 5.1 交互方式

点击列表“详情”打开右侧抽屉，建议宽度 70%。详情只读，并按流水类型动态展示关联区域。

### 5.2 基本信息

所有流水均展示：

- 积分订单号；
- 会员姓名、会员编号、手机号；
- 积分规则编码和规则名称；
- 积分方向和操作类型；
- 基础积分、倍数和最终积分；
- 操作前积分、操作后积分；
- 冻结前积分、冻结后积分；
- 可用状态和订单状态；
- 业务类型和业务单号；
- 原积分订单号；
- 操作人、操作渠道、操作时间；
- 备注或计算快照。

### 5.3 入账批次

增加积分流水通过以下关系查询：

```text
xhbs_points_user_into_bill_detail.order_no_src
= xhbs_points_order.order_number
```

展示字段：

| 字段 | 说明 |
|---|---|
| 入账批次 ID | `id` |
| 原始积分 | `points` |
| 已核销积分 | `points - remaining_points` |
| 剩余积分 | `remaining_points` |
| 生效时间 | `created_at`，冻结积分在解冻时创建批次 |
| 到期时间 | `expired_time` |
| 批次状态 | 未消费、部分消费、全部消费、已过期、已冲销 |

### 5.4 出账记录

扣减、消费、过期和冲销流水通过以下关系查询：

```text
xhbs_points_user_out_bill_detail.order_no_src
= xhbs_points_order.order_number
```

展示字段：

- 出账记录 ID；
- 出账积分；
- 出账原因；
- 订单号；
- 状态；
- 创建时间。

### 5.5 批次核销明细

通过出账记录查询核销关系：

```text
xhbs_points_user_out_bill_detail.id
= xhbs_points_user_bill_detail.out_bill_id

xhbs_points_user_bill_detail.into_bill_id
= xhbs_points_user_into_bill_detail.id
```

展示字段：

- 入账批次 ID；
- 入账订单号；
- 积分来源；
- 批次生效时间；
- 批次到期时间；
- 本次核销积分；
- 核销前批次剩余积分；
- 核销后批次剩余积分。

现有核销表未保存核销前后批次余额，一期可以通过订单和当前批次展示；如业务要求保存不可变快照，再在核销表增加快照字段，不新建表。

## 7. FIFO 消费设计

### 7.1 可消费批次

所有人工减积分、礼品兑换、活动抵扣及其他积分消费必须调用统一积分服务，并在事务中执行：

```sql
SELECT *
FROM xhbs_points_user_into_bill_detail
WHERE member_id = #{memberId}
  AND account_type = 0
  AND is_del = 0
  AND bill_status IN (0, 1)
  AND remaining_points > 0
  AND expired_time > NOW()
ORDER BY created_at ASC, id ASC
FOR UPDATE;
```

所有批次有效期相同，按生效时间排序即为先进先出。冻结积分未解冻前不得进入可消费批次；解冻时创建入账批次。

### 7.2 跨批次消费

消费 8 积分时，若批次 A 剩余 5、批次 B 剩余 20：

```text
批次 A 扣 5，remaining_points = 0，状态改为全部消费；
批次 B 扣 3，remaining_points = 17，状态改为部分消费；
同一张出账单写入两条 xhbs_points_user_bill_detail。
```

必须满足：

```text
out_bill.points
= SUM(points_user_bill_detail.points WHERE out_bill_id = 当前出账记录)
```

### 7.3 事务与锁顺序

统一锁顺序：

```text
锁定 member
→ 锁定可消费入账批次
→ 创建积分订单
→ 创建出账记录
→ 更新入账批次
→ 创建核销明细
→ 更新会员余额
```

禁止消费后再异步更新入账批次，避免会员余额和批次余额不一致。

## 8. 360 天过期设计

### 8.1 有效期口径

积分从“可消费生效时间”起按命中规则的 `points_valid_days` 计算，默认 360 天：

```sql
expired_time = DATE_ADD(effective_time, INTERVAL points_valid_days DAY)
```

- 普通积分即时生效，入账批次 `created_at` 即生效时间；
- 购卡积分冻结 7 天，解冻并创建可用批次时才开始计算有效期；
- 新增规则默认 360 天，规则可在 1~3650 天范围内调整；修改规则只影响后续新入账批次；
- `expired_time <= NOW()` 时视为已过期。

### 8.2 过期任务

Admin 模块增加积分过期任务，建议每 10 分钟执行一次，每批最多处理 200 条：

```sql
SELECT id
FROM xhbs_points_user_into_bill_detail
WHERE is_del = 0
  AND account_type = 0
  AND bill_status IN (0, 1)
  AND remaining_points > 0
  AND expired_time <= NOW()
ORDER BY expired_time ASC, id ASC
LIMIT 200;
```

每个入账批次使用独立事务：

1. 锁定会员；
2. 锁定入账批次并再次检查到期条件；
3. 读取当前 `remaining_points`；
4. 创建 `EX` 开头的负数积分订单；
5. 创建积分过期出账记录；
6. 创建出账记录与入账批次的核销明细；
7. 将入账批次 `remaining_points` 更新为 0，状态更新为已过期；
8. 会员 `current_points` 只扣除实际剩余积分；
9. 同一事务提交。

示例：入账 10 积分，已消费 5 积分，到期任务只创建 `-5` 的过期流水。

### 8.3 幂等

积分过期订单业务键：

```text
POINTS_EXPIRE:<into_bill_id>
```

复用 `xhbs_points_order.business_key` 唯一索引，保证任务重试不会重复扣减。

### 8.4 消费入口兜底

即使定时任务延迟，消费查询也必须排除 `expired_time <= NOW()` 的批次。消费前应同步结算该会员已到期批次，确保已过期积分不能继续使用。

## 11. 数据一致性

系统必须保持：

```text
member.current_points
= SUM(xhbs_points_user_into_bill_detail.remaining_points)
```

统计范围：当前会员、常规账户、未删除、已生效且未过期批次。

禁止在不维护批次的情况下直接修改 `member.current_points`。如发现会员余额与批次余额不一致，应记录对账异常，不得使用 `MIN(余额, 过期积分)` 静默修正。

## 12. 历史数据迁移

现有扣积分逻辑没有核销入账批次，历史 `remaining_points` 不能直接视为真实余额。启用 FIFO 前必须执行一次数据核对：

1. 按会员比较 `member.current_points` 与有效入账批次余额合计；
2. 输出不一致会员清单；
3. 对不能还原的历史余额创建“历史积分期初批次”；
4. 历史期初批次的有效期由业务确认，推荐从新规则上线日起计算 360 天；
5. 迁移后再次核对会员余额与批次余额。

未经核对不得直接启用过期任务，否则可能重复过期或误扣历史积分。

## 13. 验收标准

1. 积分流水可以按会员、规则、方向、类型、状态和时间分页查询；
2. 店员不能查看其他门店会员积分；
3. 详情能展示积分总流水及关联入账、出账和核销明细；
4. 一次扣减跨多个入账批次时，按批次生效时间先进先出；
5. 每条核销记录都能关联一条入账批次和一条出账记录；
6. 入账 10、消费 5 后，到期只扣减剩余 5；
7. 冻结积分在解冻前不可消费，解冻后按原发放规则快照开始计算有效期；
8. 过期任务重复执行不会重复扣减；
9. 会员余额、入账批次余额、出账记录和核销明细保持一致；
10. 页面不提供修改和删除历史积分流水的入口。
