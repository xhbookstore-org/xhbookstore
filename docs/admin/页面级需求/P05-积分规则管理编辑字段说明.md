# P05 - 积分规则管理编辑字段说明

## 1. 页面用途

积分规则管理用于配置会员积分的获取、消耗和人工调整规则。规则由业务场景和触发事件定位，由计算模式决定积分值，再由适用策略限制生效范围、次数和预算。

规则修改只影响后续新业务。已经生成的积分流水保存规则快照，不会因规则后续修改而改变。已有流水的规则不允许直接删除，通常应将运行状态改为“停用”。

## 2. 基础信息

| 字段 | 示例/选项 | 含义和作用 |
|---|---|---|
| 规则编码 `ruleCode` | `BORROW_BOOK` | 规则的稳定唯一标识，供业务代码匹配。创建后不可修改；建议使用大写字母、数字和下划线。 |
| 规则名称 `ruleName` | 借阅图书奖励 | 给管理员看的业务名称，用于列表、查询和审计。 |
| 业务场景编码 `sceneCode` | `BORROW_BOOK` | 标识规则适用的业务场景，例如注册、购卡、借阅、退卡。应与业务服务传入的场景编码一致。 |
| 触发事件 `triggerEvent` | `BORROW_ORDER_CREATED` | 标识何时尝试执行规则，例如订单支付、借阅单创建。自动规则由系统事件触发。 |
| 规则来源 `ruleSource` | 系统 / 自定义 | 区分系统内置规则和运营人员创建的规则。 |
| 开发状态 `implementationStatus` | 已有 / 开发中 / 暂无 | 表示对应业务链路是否已经接入，不等于规则是否启用。 |
| 运行状态 `status` | 草稿 / 启用 / 停用 / 结束 | 控制规则当前能否被匹配。只有启用状态且在有效期内的规则可参与积分计算。 |

## 3. 积分计算

| 字段 | 选项/单位 | 含义和作用 |
|---|---|---|
| 积分方向 `direction` | 获取 `ADD` / 消耗 `DEDUCT` | 决定积分进入还是扣出会员账户。冻结天数只适用于获取积分。 |
| 触发方式 `triggerMode` | 自动 `AUTO` / 手动 `MANUAL` | 自动由业务事件触发；手动由员工或管理后台提交。 |
| 计算方式 `calculationMode` | 固定积分 / 按数量 / 按金额 / 人工输入 / 按原订单 | 决定积分的计算来源。选择后，页面只展示相关参数。 |
| 固定积分 `fixedPoints` | 正整数 | 固定模式下每次业务产生的积分值。启用固定规则时必须填写。 |
| 单位积分 `pointsPerUnit` | 正整数 | 按数量或金额模式使用。按数量表示每本/每件积分；按金额表示每 1 元对应的积分。 |
| 冻结天数 `freezeDays` | 0-3650 天 | 获取积分到账后冻结的天数。为 0 表示立即可用；消耗规则不能设置冻结。 |

计算模式说明：

- 固定积分：`fixedPoints`。
- 按数量：`业务数量 × pointsPerUnit`，单位类型为册/件。
- 按金额：`金额 × pointsPerUnit`，单位通常为元，具体取整规则由业务服务执行。
- 人工输入：积分由操作员输入；系统会强制打开“允许修改积分”。
- 按原订单：用于退货、退卡等冲正场景，按原积分订单实际值处理，不填写固定值或单位值。

## 4. 适用策略

| 字段 | 类型/示例 | 含义和作用 |
|---|---|---|
| 必须业务单号 `requireBizOrder` | 开关 | 是否必须绑定业务订单号。建议开启，用于幂等控制和追溯。 |
| 必须凭证 `requireEvidence` | 开关 | 预留的凭证要求标记，当前仅保存配置和供后续业务扩展使用，尚未在积分执行接口中强制校验上传凭证或备注。 |
| 排除批量采购 `excludeBulkPurchase` | 开关 | 开启后，批量采购类订单不使用该规则，避免批量业务误发积分。 |
| 启用会员日 `memberDayEnabled` | 开关 | 是否在指定会员日对积分进行倍增。开启后必须填写会员日日期和倍数。 |
| 会员日日期 `memberDayDays` | `[6,16,26]` | 每月参与会员日计算的日期列表，取值范围 1-31，系统会去重并排序。 |
| 会员日倍数 `memberDayMultiplier` | 例如 `2` | 会员日积分乘数，必须大于 0。 |
| 允许修改积分 `manualPointsEditable` | 开关 | 是否允许人工调整计算出的积分。人工输入模式会自动开启；自动规则应谨慎开启。 |
| 单会员次数 `memberLimit` | 次数 | 单个会员最多命中该规则的次数。留空或 0 表示不限制。 |
| 规则总次数 `totalLimit` | 次数 | 所有会员合计最多命中次数。留空或 0 表示不限制。 |
| 单笔积分上限 `maxPointsPerOrder` | 积分 | 单笔业务最多产生的积分，防止异常金额或数量导致过度发放。留空或 0 表示不限制。 |
| 积分预算 `budgetPoints` | 积分 | 该规则可发放的总积分预算。留空或 0 表示不限制。 |
| 生效时间 `effectiveFrom` | 日期时间 | 规则开始参与匹配的时间，留空表示立即生效。 |
| 结束时间 `effectiveTo` | 日期时间 | 规则停止参与匹配的时间，留空表示长期有效；不能早于生效时间。 |
| 排序 `sortOrder` | 非负整数 | 管理列表和人工积分规则候选的排序值，数值越小越靠前。当前自动积分匹配不会使用该字段决定最终规则。 |
| 备注 `remark` | 最多 500 字 | 记录配置原因、运营说明、适用范围或审批信息。 |

## 5. 配置注意事项

1. 规则编码、业务场景编码和触发事件必须与业务代码约定一致，否则规则保存成功也不会被业务匹配。
2. 启用规则前必须完成对应积分计算参数配置；固定模式填写固定积分，按数量/金额模式填写单位积分。
3. 获取积分才允许设置冻结天数；消耗积分、按原订单冲正不应设置冻结。
4. 会员日开关关闭时，日期和倍数不会生效；开启后日期必须是 1-31 的数组，倍数必须大于 0。
5. 次数上限、单笔上限和预算均为保护性限制，留空或 0 代表不限制，不代表禁止发放。
6. 停用或结束规则不会修改历史积分流水；需要调整规则时，优先修改后续生效时间或新建版本。

当前自动规则匹配按 `sceneCode`、`triggerEvent`、部门、会员卡类型和有效期筛选，最终按 `effectiveFrom` 倒序、`id` 倒序取一条规则；没有按 `sortOrder` 做自动优先级选择。人工积分规则候选列表则按 `sortOrder`、`id` 升序展示，最终由操作员选择具体规则。

## 6. 当前实现状态

`requireEvidence` 当前已完成前端开关、Java 实体字段、数据库字段和新增/修改接口的读写，但积分发放、扣减和人工录入服务尚未使用该字段进行校验。因此：

- 开启后不会自动弹出凭证上传控件；
- 请求未携带凭证或备注时不会因此失败；
- 当前它只能作为规则配置记录，不能作为实际的强制凭证策略。

如需真正启用，应在人工积分、消费录入等入口增加凭证字段，并在积分服务执行前校验凭证是否存在，同时将凭证地址或凭证编号写入积分订单和审计日志。
